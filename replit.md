## VoiceIntel Features
- **Local Whisper transcription** via faster-whisper (no cloud APIs)
- **IMAP email ingestion** — polls for voicemail attachments (configurable interval)
- **NLP pipeline** — keywords, sentiment, urgency detection, category classification
- **Analytics dashboard** — 30-day trend, sentiment donut, category bars, keyword cloud, hourly distribution, urgency keywords, frequent callers
- **Frequent Callers card** (analytics page, third tile in the "Top Keywords / Urgency Keywords / Frequent Callers" row) — groups all in-scope voicemails by parsed phone number (digits-only key, so `(555) 123-4567` and `555.123.4567` cluster) and shows the top 10 numbers ranked by total voicemails, with most-recent date as tiebreaker. Each row shows the caller's name (using `Voicemail.display_caller_name` so a single AI-extracted name from any of the grouped voicemails wins over a generic carrier label), the formatted phone, the last-received date in DISPLAY_TZ, the total count, and a horizontal bar normalised to the top entry. The whole row links to `/voicemails?q=<digits>` for drill-down. Computed in `main.py:analytics()` with `selectinload(Voicemail.insights)` so accessing `display_caller_name` (which reads `vm.insights.ai_caller_name`) doesn't N+1 across hundreds of rows; scoped via `scope_voicemails` like all other analytics aggregates. Voicemails whose subject doesn't yield a parseable phone are skipped (no stable identifier). The row layout uses a new `.grid-1-1-1` utility class added to `base.html` (3-col → 2-col below 1100px → 1-col below 720px).
- **AI Insights** — narrative analytics summary generated by Phi-3 mini (local, via Ollama). Runs **once per hour** as a background APScheduler job; results cached in `analytics_insights` table and rendered server-side. No on-demand button — page just shows the latest cached output.
- **Per-voicemail AI summary** — after transcription completes, `ai_summary_service` calls Phi-3 mini (Ollama at `OLLAMA_BASE_URL`, default `http://localhost:11434/v1`) on each transcript and stores a structured result on the `insights` row: `ai_summary` (2-3 sentences), `ai_intent` (short phrase), `ai_action_items` (JSON array, max 5), `ai_suggested_response` (draft reply), `ai_caller_name` (VARCHAR(120), the caller's spoken name if they introduce themselves in the transcript), plus bookkeeping (`ai_status`, `ai_error`, `ai_duration_ms`, `ai_generated_at`). Surfaces as an "AI Summary" card on the voicemail detail page above the transcript with a Regenerate button (POST `/voicemails/<id>/ai-summary`, JSON response, inline DOM update — no full reload). Generation is best-effort: a model timeout/outage logs a warning and marks `ai_status='error'` but never blocks the voicemail itself from being saved or shown. Transcripts are capped at 4000 chars before being sent. Output is parsed from a labeled-text format (`CALLER NAME:` / `SUMMARY:` / `INTENT:` / `ACTION ITEMS:` / `SUGGESTED RESPONSE:`) rather than JSON because Phi-3 mini is more reliable with prose. The CALLER NAME parser (`_normalise_caller_name`) rejects placeholder tokens the model emits when no name is given (`Unknown`, `N/A`, `Anonymous`, `The Caller`, etc.) and strips quotes/brackets/trailing punctuation; multi-line responses are truncated to the first line and length-capped at 120.
- **Spoken-name override (display_caller_name)** — many carriers send a useless caller-ID like `WIRELESS CALLER`, `ANONYMOUS`, or a city/state (`LOS ANGELES CA`) in the SendGrid subject line, even when the caller introduces themselves in the message ("Hi, my name is David Gorman…"). `Voicemail.display_caller_name` returns the carrier-supplied name when it looks like a real person/business, otherwise falls back to `insights.ai_caller_name` extracted by Phi-3. The "generic" detector (`_is_generic_caller_name` in `app/models/voicemail.py`) covers a fixed allow-list (`Wireless Caller`, `Unknown`, `Anonymous`, `Restricted`, `Private`, `No Caller ID`, `Out of Area`, `Spam Risk`, `Scam Likely`, etc.) plus a `(City) (State)` pattern that fires when the last token is a 2-letter US state code (handles `Los Angeles Ca`, `Houston Tx`, `Phoenix Az`, …). When the AI name is shown in place of the carrier name, `Voicemail.carrier_caller_label` returns the original carrier label so the templates can render it as a small dim parenthetical underneath/next to the name (`David Gorman (Wireless Caller)`) — the carrier label is never lost, just demoted. Used by `voicemails.html` (table cell), `voicemail_detail.html` (header + page title + breadcrumb), and `tasks.html` (task row link). `Voicemail.to_dict` exposes all three: `caller_name` (the chosen display name), `carrier_caller_name` (raw carrier value), and `ai_caller_name` (raw AI value). No backfill is needed — existing voicemails will pick up an AI name the next time they are summarised, and the existing per-voicemail Regenerate button is the manual path.
- **Display timezone** — all timestamps are stored as UTC (`datetime.utcnow()`) but rendered through the `localtime` Jinja filter (`{{ dt | localtime('%b %d, %Y %H:%M %Z') }}`) registered in `app/__init__.py`. Defaults to `America/Chicago`; override via `DISPLAY_TZ` env var. Use `%Z` in the format string to include the abbreviation (CDT/CST). The configured zone name is also exposed to JS templates via `DISPLAY_TZ_NAME` for `Intl.DateTimeFormat`-based formatters (e.g. the inline AI Summary timestamp update on regenerate).
- **Full-text search** across all transcripts
- **Voicemail detail** — audio player, full transcript, segments timeline, insights, callbacks, notes
- **Roles** — `admin`, `supervisor`, `agent`, `viewer`. Admins have full access; supervisors manage non-admin users + assign callbacks; agents receive callback assignments + can add notes; viewers are read-only but can still add notes. Supervisors are scoped to their own teams in `/admin/*`: they only see/manage users in teams they belong to, can only invite or directly create `agent`/`viewer` accounts (never supervisor/admin), can only assign invitees to their own teams, can only see invitations they themselves sent (sidebar badge & list both filtered by `invited_by_id`), and cannot remove or manage admin/supervisor peers. Team create/delete remain admin-only.
- **Email invitations** — admins/supervisors send invites at `/admin/invites`. Recipient gets a SendGrid email with a one-time link (`/invite/<token>`) where they choose their own password and (optionally) edit the suggested name. Accepted invites create the User row + apply optional team memberships in one step. Tokens are 256-bit `secrets.token_urlsafe(32)` values, expire after 7 days, and a fresh invite to the same email auto-revokes any prior pending one. Status (`pending`/`accepted`/`expired`/`revoked`) is derived from `accepted_at` / `revoked_at` / `expires_at` rather than stored, eliminating drift. Sidebar shows a pending-invite badge for user-managers. Email links use `APP_BASE_URL` env var (recommended for self-hosted) and fall back to the request scheme + Host.
- **Teams** — agents belong to one or many teams (M:N). Incoming voicemails are auto-routed to a team via per-team rules: `recipient_email`/`recipient_domain` (uses the SendGrid Inbound-Parse "to" field — e.g. `team1@mail3.opscal.io` → Sales), `sender_email`/`sender_domain`, `keyword` (substring of transcript or subject), or `caller_phone` (digit substring). Rules evaluate lowest-priority-number first, first match wins. Routing runs twice per voicemail: once on initial save (recipient/sender/phone) and again after transcription (gives keyword rules a chance). **Team scoping (enforced everywhere):** agents/viewers/**supervisors** see only voicemails routed to their teams **plus** unrouted voicemails; only admins see everything. The same scope is applied to voicemails list/detail/audio/notes/delete/status, dashboard, all analytics aggregates, the cached AI Insights card (admin-only), `/voicemails/poll`, `/api/stats`, `/api/voicemails`, `/api/search`, `/api/voicemails/<id>` + `/reprocess`, the tasks inbox (both `mine` and `all` views, joined to Voicemail), callback create/update/delete + `_can_modify_callback` (assignees lose access if the underlying voicemail moves out of scope), team override dropdowns and bulk team-assign (uses `with_for_update()` for race safety), and the assignable-user dropdown for callbacks (self + active agents on shared teams). Supervisors/admins can manually override the team on the voicemail detail page — manual changes set `team_locked=True` so auto-routing won't overwrite them. **Routing rule CRUD is admin-only** (a supervisor with rule-write access could otherwise create a broad/high-priority rule to siphon foreign voicemails into their team). Managed at `/admin/teams`.
- **Callbacks** — supervisors/admins assign follow-up phone calls to agents. Each callback has status (pending/in_progress/completed/cancelled), priority (normal/urgent), optional due date and instructions. Visible on the voicemail detail page and in the per-user "My Tasks" inbox at `/tasks`.
- **Notes & disposition** — anyone signed in can add free-form notes to any voicemail. Author or supervisor/admin can delete.
- **Soft-delete + Deleted folder** — `voicemails.deleted_at` (indexed) + `deleted_by_id` (FK users, SET NULL) added by the boot guard `_ensure_voicemails_columns`. Per-row delete (`POST /voicemails/<id>/delete`, supervisor+admin) and bulk delete (`POST /voicemails/bulk/delete`, supervisor+admin, scoped + `with_for_update()`) both *soft*-delete — they set `deleted_at` instead of removing rows, so audio + transcript + insights + callbacks + notes are all preserved for restore. `scope_voicemails` and `can_view_voicemail` filter `deleted_at IS NULL` by default (passed `include_deleted=True` only by the admin Deleted-folder routes), so soft-deleted voicemails vanish from every list/detail/analytics/API/callbacks/poll endpoint for everyone — including admins — outside the trash. The dashboard insight-keywords path was unified to always go through `scope_voicemails` so the deleted filter is honoured for admins too. `insights_service` (cached AI insights) also excludes deleted rows from totals/sentiment/categories/keywords/snippets. **Admin-only Deleted folder** at `/voicemails/deleted` with `POST /voicemails/<id>/restore` (clears `deleted_at`) and `POST /voicemails/<id>/purge` (hard-delete: cascades + removes audio files, refuses unless the row is already in trash). Sidebar shows a "Deleted" link to admins only. Bulk-delete UI lives in the existing sticky action bar on `/voicemails` — separated `can_bulk_assign` (needs assignable teams) vs `can_bulk_delete` (admin/supervisor) so a supervisor with no team-assign options still gets the Delete button. The delete button mirrors the row checkboxes into a hidden sibling form (`#bulk-delete-form`) at submit time to share selection with the team-assign form without nesting. **Cross-cutting consistency** (architect-driven follow-ups): (a) `tasks.py` always joins `Voicemail` + `scope_voicemails`, so soft-deleted callbacks disappear from `/tasks` for admins too — not just supervisors/agents. (b) The base-template `open_task_count` sidebar badge does the same join + scope so the badge always agrees with `/tasks`. (c) `voicemail_purge` resolves relative `original_path`/`converted_path` against `dirname(app.root_path)` (mirroring `serve_audio`) before unlinking and logs misses, preventing orphaned audio when the worker cwd differs. (d) `pipeline.process_email_items` has THREE `db.session.refresh(voicemail)` + `if deleted_at: continue` guards — one immediately after the initial create commit (skips first-pass routing, transcription, NLP, second commit, AI summary, second-pass routing, triggers), one after the second commit (before the AI summary block), and one after AI summary completes (before second-pass routing + `run_triggers`, closing the longest TOCTOU window since Phi-3 can take 10-30s). All three treat `refresh()` failure as "skip" so the worker never proceeds against a row whose state it can't confirm. (e) `_can_modify_callback` (covers `POST /tasks/<id>/update`) and `delete_callback` (`POST /tasks/<id>/delete`) both refuse — for everyone, including admins — when the underlying voicemail is soft-deleted, enforcing the "trashed voicemails are only mutable through restore/purge" invariant. `create_callback` is already covered because `can_view_voicemail` filters `deleted_at IS NULL` even for admins.
- **Background scheduler** via APScheduler

## Environment Variables (for VoiceIntel)
| Variable | Description | Default |
|---|---|---|
| `DATABASE_URL` | PostgreSQL connection URL | (set as secret) |
| `IMAP_HOST` | IMAP server hostname | — |
| `IMAP_PORT` | IMAP port | 993 |
| `IMAP_USERNAME` | Email username | — |
| `IMAP_PASSWORD` | Email password | — |
| `IMAP_FOLDER` | Folder to monitor | INBOX |
| `WHISPER_MODEL` | tiny/base/small/medium/large-v2 | base |
| `POLL_INTERVAL` | Email poll interval (seconds) | 60 |
| `APP_BASE_URL` | Public base URL for emailed invite links (e.g. `https://voice-ai.obtv.io`); falls back to request scheme + Host if unset | (empty) |
| `OLLAMA_BASE_URL` | OpenAI-compatible base URL for the local Ollama instance serving Phi-3 mini (used by both AI Insights and per-voicemail summaries) | `http://localhost:11434/v1` |
| `DISPLAY_TZ` | IANA timezone for displayed timestamps (e.g. `America/Los_Angeles`, `Europe/London`). All data is stored in UTC; only the rendered output is converted. Invalid values fall back to `UTC` with a startup warning. | `America/Chicago` |

```
  requirements.txt
```

### Schema migrations
The app uses `db.create_all()` for first-boot table creation (this picks up
brand-new tables like `user_invites` automatically). New columns added to
existing tables are applied via idempotent boot guards in `app/__init__.py`:
- `_ensure_voicemails_columns()` — adds `recipient`, `team_id`, `team_locked`
  (Teams feature) to the `voicemails` table.
- `_ensure_insights_columns()` — adds the eight `ai_*` columns
  (per-voicemail AI summary feature) to the `insights` table.

Both are safe on Postgres and SQLite. Add new columns to the matching helper
when extending one of those tables again, or introduce a real Alembic
migration if the schema keeps growing.

## Stack (Node.js/TypeScript monorepo)

## Key Commands

- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
